    **constants.ts** -- Update TUS_RETRY_DELAYS for exponential backoff:
    - Change `[0, 1000, 3000, 5000]` to `[1000, 2000, 4000]` (3 retries, exponential, matches RESIL-01 requirement of 3 attempts).
    - Keep all other constants unchanged.

    **tusUpload.ts** -- Enable resume and add smart retry:
    - Change `storeFingerprintForResuming: false` to `storeFingerprintForResuming: true`. Remove the Phase 9 TODO comment.
    - Add `onShouldRetry` callback to the tus.Upload options. The callback receives `(err, retryAttempt, options)`:
      - Extract HTTP status: `const status = err.originalResponse?.getStatus?.() ?? 0;`
      - Never retry: 413 (too large), 415 (unsupported type), 410 (expired), 403 (forbidden) -- return false.
      - Always retry: 0 (network error), 408 (timeout), 429 (rate limited), 502/503/504 (proxy errors) -- return true.
      - Default: `!(status >= 400 && status < 500) || status === 409 || status === 423`.
    - Add an optional `onBeforeRetry?: () => void` field to the `TusUploadCallbacks` interface. This callback fires when a retry is about to happen (before the delay).
    - In the `onShouldRetry` callback: when the decision is to retry (returning true), call `callbacks.onBeforeRetry?.()` before returning true. This lets the hook layer signal "retrying" to the UI.
    - Keep the function signature and return type unchanged. Keep `removeFingerprintOnSuccess: true`.

    **tusErrorClassifier.ts** -- Create new module:
    - Export interface `ClassifiedUploadError` with fields: `userMessage: string`, `technicalDetail: string`, `isRetryable: boolean`.
    - Export function `classifyUploadError(error: Error): ClassifiedUploadError`.
    - Cast error to access `originalResponse?.getStatus()` and `originalResponse?.getBody()` (tus-js-client DetailedError shape).
    - Classification map:
      - status 0 / no response: "Network connection lost. Check your internet and try again." (retryable)
      - 413: "File exceeds the maximum upload size." (not retryable)
      - 410: "Upload session expired. Please start the upload again." (retryable -- retry creates new upload)
      - 403: "Upload not permitted. The server rejected this request." (not retryable)
      - 415: "File type not supported by the server." (not retryable)
      - 500-599: "Server error occurred. Please try again in a moment." (retryable)
      - 520-524 (Cloudflare): "Server connection issue. Please try again." (retryable)
      - Fallback: "Upload failed. Please try again." (retryable)
    - Include HTTP status and response body in technicalDetail for all cases.
    - Keep this module pure (no React imports, no side effects).
  </action>

    2. Change the hook-level `TusUploadCallbacks` interface:
       - `onError` signature changes from `(error: string) => void` to `(userMessage: string, technicalDetail: string, isRetryable: boolean) => void`.
       - Add `onRetrying?: () => void` -- fires when a retry is about to happen.
       This gives the orchestration layer the classified error data and retry signals directly.

    3. `startTusUpload` stays synchronous -- it returns `{ abort: () => void }` immediately:
       - `abort` captures the `upload` instance via closure, so it works even before `start()` is called.
       - After creating the upload via `createTusUpload()`, kick off the async resume+start flow internally (no await needed by caller):
         ```
         const upload = createTusUpload(file, metadata, { ... });
         // Fire-and-forget async resume check + start
         (async () => {
           const previousUploads = await upload.findPreviousUploads();
           if (previousUploads.length > 0) {
             upload.resumeFromPreviousUpload(previousUploads[0]);
           }
           upload.start();
         })();
         return { abort: () => upload.abort(true) };
         ```
       - This preserves the synchronous return signature. The orchestration layer does NOT need to await.

    4. In the `onError` callback passed to `createTusUpload`:
       - Call `const classified = classifyUploadError(error)`.
       - Call `callbacks.onError(classified.userMessage, classified.technicalDetail, classified.isRetryable)`.

    5. Pass `onBeforeRetry: callbacks.onRetrying` through to `createTusUpload` (Task 1 already added `onBeforeRetry` to the library-level `TusUploadCallbacks` interface and wired it in `onShouldRetry`). The hook just forwards the callback.

         currentTaskIdRef.current = null;
       }
