### Section 1: Overview

Bullet list:
- What the migration does (replaces `Base.metadata.create_all()` with Alembic; adds `users`, `api_keys`, `subscriptions`, `usage_events`, `rate_limit_buckets`, `device_fingerprints`; tightens `tasks.user_id` to NOT NULL)

  | Revision | Purpose |
  |----------|---------|
  | `0001_baseline` | Empty-stamp baseline matching v1.1 `tasks` shape |
  | `0002_auth_schema` | Six new auth/billing tables + `tasks.user_id` (nullable) |
  | `0003_tasks_user_id_not_null` | Tightens `tasks.user_id` NOT NULL + adds `idx_tasks_user_id` |

- Pre-requisite checklist:

  - `uv sync` has run; `.venv` is active
  - `.env` has `DB_URL` pointing at the production records.db
  - Operator has shell-level access (not just `docker exec` — getpass needs a real TTY in step 5)

### Section 2: Pre-flight: Backup Database

Single fenced bash block:

```bash
# Stop the application (or take it out of rotation) before backing up.
# Replace `records.db` with the path your DB_URL points at.
cp records.db records.db.pre-v1.2.bak

# Verify the backup is non-empty and readable.
sqlite3 records.db.pre-v1.2.bak "SELECT COUNT(*) FROM tasks;"
```

Brief note: the rollback section assumes this backup file is present. Do not skip.

### Section 3: Stamp Baseline (alembic stamp 0001_baseline)

Explanation: brownfield records.db already has a `tasks` table, so we stamp the chain at 0001 (do NOT run `upgrade` from base — that would try to recreate `tasks` and fail).

```bash
# From project root with .venv activated.
uv run alembic stamp 0001_baseline

# Verify the stamp landed.
uv run alembic current
# Expected output: "0001_baseline (head)"
```

### Section 4: Upgrade Auth Schema (alembic upgrade 0002_auth_schema)

Explanation: this revision creates six new tables and adds the nullable `user_id` FK to `tasks`. Existing rows are preserved; `tasks.user_id` is NULL on every pre-existing row.

```bash
uv run alembic upgrade 0002_auth_schema

# Confirm the new tables exist.
sqlite3 records.db ".tables"
# Expected (alphabetical): alembic_version api_keys device_fingerprints
# rate_limit_buckets subscriptions tasks usage_events users

# Confirm tasks gained the user_id column (nullable for now).
sqlite3 records.db "PRAGMA table_info(tasks);" | grep user_id
# Expected: <ordinal>|user_id|INTEGER|0||0
```

### Section 5: Create Admin User

Explanation: the backfill step needs a real user row to reassign orphan tasks to. Create the admin BEFORE backfilling. The CLI prompts for the password TWICE via `getpass`; never echoes it; never accepts it as a flag.

```bash
# Replace the email with a real operator address.
uv run python -m app.cli create-admin --email admin@example.com

# Prompts:
#   Admin password: <type, no echo>
#   Confirm password: <type, no echo>
# On success: "Admin user <id> created with email admin@example.com"
# Exit code: 0 on success; 1 on password mismatch / duplicate email / weak password.
```

Platform note (subsection — NOT a new top-level section):

> **Windows / non-TTY note:** `getpass.getpass()` requires a real terminal. Piping the password via `echo "pw\npw" |` does NOT work on Windows (`msvcrt.getwch()` reads keyboard directly). Run this command in an interactive shell. For automated pipelines see the `tests/integration/test_phase12_cli_backfill_e2e.py` test-only `-c` preamble pattern; do NOT modify production source.

### Section 6: Backfill Tasks

Explanation: every `tasks` row created under v1.1 has `user_id IS NULL`. The 0003 migration's pre-flight guard refuses to run while orphans exist. This step reassigns them.

```bash
# Dry-run first to see how many rows would change.
uv run python -m app.cli backfill-tasks --admin-email admin@example.com --dry-run
# Output: "Would reassign <N> orphan tasks to admin admin@example.com (id=<id>). [dry-run]"

# Then commit. --yes skips the y/N prompt for scripted runbooks.
uv run python -m app.cli backfill-tasks --admin-email admin@example.com --yes
# Output: "Reassigned <N> orphan tasks to admin admin@example.com."
# Exit code: 0 on success; 1 if post-condition COUNT(user_id IS NULL) != 0.
```

Idempotency note: re-running with 0 orphans exits 0 with the message `No orphan tasks to backfill.` Safe to retry.

### Section 7: Upgrade Head (Apply NOT NULL)

Explanation: the final migration tightens `tasks.user_id` to NOT NULL and creates `idx_tasks_user_id`. The pre-flight guard re-checks `SELECT COUNT(*) FROM tasks WHERE user_id IS NULL` and refuses to alter the column if non-zero.

```bash
uv run alembic upgrade head

# Confirm the chain landed at 0003.
uv run alembic current
# Expected output: "0003_tasks_user_id_not_null (head)"
```

If the guard fires (you skipped step 6):

```text
RuntimeError: Refusing to apply 0003_tasks_user_id_not_null:
<N> tasks have user_id IS NULL. Run
`python -m app.cli backfill-tasks --admin-email <e>` first.
```

Recovery: run step 6, then re-run this command. The migration is idempotent at this boundary.

### Section 8: Smoke Verify

Quick post-migration checks. All should pass after the upgrade.

```bash
# 1. Migration chain landed at head.
uv run alembic current
# Expected: "0003_tasks_user_id_not_null (head)"

# 2. tasks.user_id is NOT NULL on every row.
sqlite3 records.db "SELECT COUNT(*) FROM tasks WHERE user_id IS NULL;"
# Expected: 0

# 3. The idx_tasks_user_id index exists.
sqlite3 records.db "SELECT name FROM sqlite_master WHERE type='index' AND tbl_name='tasks';"
# Expected (set; order varies): idx_tasks_user_id sqlite_autoindex_*

# 4. Admin user exists with plan_tier='pro'.
sqlite3 records.db "SELECT id, email, plan_tier FROM users WHERE email='admin@example.com';"
# Expected: <id>|admin@example.com|pro

# 5. App boots cleanly (Alembic is the schema source of truth — Base.metadata.create_all is gone from main.py).
uv run uvicorn app.main:app --port 8000
# Expected: startup logs show "Application startup complete." with no schema errors.
# Stop with Ctrl+C; the v1.2 frontend in /ui will require login next.
```

### Section 9: Rollback

Two rollback paths. Choose based on how far the migration progressed.

**A. Reversible step-back via Alembic (no data loss for unrelated tables):**

```bash
# Reverse the most recent migration.
uv run alembic downgrade -1

# Reverse step-by-step further if needed.
uv run alembic downgrade 0002_auth_schema  # back out 0003
uv run alembic downgrade 0001_baseline     # back out 0002 (drops 6 auth tables; tasks.user_id col removed)
uv run alembic downgrade base              # only on a freshly-created (greenfield) DB; drops `tasks`
```

Note: `alembic downgrade -1` from `0003` reverses the NOT NULL + index. Going further (`0002 -> 0001`) drops the six auth tables and removes the `tasks.user_id` column. Backfilled `tasks.user_id` values are LOST on the `0002 -> 0001` step.

**B. Full restore from backup (unrecoverable mid-migration):**

```bash
# Stop the application.
# Restore the pre-v1.2 backup over the current DB.
cp records.db.pre-v1.2.bak records.db

# Verify the restore.
sqlite3 records.db "SELECT COUNT(*) FROM tasks;"
# Expected: same row count as pre-migration.
```

After option B, the chain is back to its pre-stamp state. Re-stamping is NOT required if you are reverting to v1.1 deploy; only re-stamp if you are about to retry the v1.2 migration.
