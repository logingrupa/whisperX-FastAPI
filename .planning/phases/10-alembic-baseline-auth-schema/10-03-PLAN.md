    **Notes on choices:**
    - `op.create_table` uses inline `sa.UniqueConstraint(...)` instead of `unique=True` per-column for named constraints (predictable for downgrade + verification). Both forms emit the same DDL but named constraints are reversible by name.
    - `users.plan_tier` uses `server_default="trial"` — DB-side default. ORM also sets `default="trial"` (Python-side). Both required: ORM default for in-memory creates without flush; server_default for raw `INSERT` from migrations or external clients.
    - `token_version` `server_default="0"` likewise.
    - No `default=` for `created_at` in migration — Alembic does not run Python lambdas; the ORM populates `created_at` at `INSERT` time via `default=lambda: datetime.now(timezone.utc)`. Migration just enforces NOT NULL.
    - `task_id` FK on `usage_events` does NOT use `ondelete=CASCADE` — we keep usage records even if a task row is later removed (audit trail). Per CONTEXT §38.
    - `tasks.created_at` / `tasks.updated_at` ALTER: migrates plain DateTime to DateTime(timezone=True). For SQLite this is a no-op at the storage layer (SQLite stores DateTime as TEXT/REAL regardless), but the column type metadata changes — render_as_batch + alter_column generates the recreate-table-copy-data sequence. The Plan 02 ORM Task.created_at/updated_at use the same `_created_at_column()/_updated_at_column()` factories as the new classes, so post-upgrade ORM and DB agree.
    - Drop order in downgrade respects FK dependencies: device_fingerprints, rate_limit_buckets, usage_events, subscriptions, api_keys all reference users — drop them all before users. usage_events also has FK→tasks; tasks survives the downgrade so order is fine. tasks.user_id constraint is dropped FIRST inside the batch (before any users-referencing table is dropped) so the FK doesn't dangle.

    **Code quality bar:**
    - DRY: every table is a single flat `op.create_table(...)` call. No helper-of-a-helper layer; Alembic ops are already a DSL.
    - SRP: this revision adds schema. No data manipulation, no Python loops over tables.
    - No spaghetti / no nested ifs: zero `if` statements. Two `with op.batch_alter_table(...)` blocks at depth 1 (upgrade + downgrade).
    - Self-explanatory: `users`, `api_keys`, `idempotency_key`, `bucket_key`, `cookie_hash`, `ua_hash`, `ip_subnet`, `device_id` — no abbreviations.
    - Tiger-style: every column declares `nullable=` explicitly (no implicit nullable=True default surprises).
