## Interaction State Machines

### `DeleteAccountDialog` ‚Äî type-email confirmation

| State | Trigger | Visual | Submit button | Email input |
|-------|---------|--------|---------------|--------------|
| **idle** | Dialog opened, input empty | No alert | `disabled` (opacity-50, no pointer events ‚Äî shadcn Button default) | enabled, autoFocus, value=`''` |
| **typing-mismatch** | input non-empty AND `lower(input) !== lower(user.email)` | No alert (do not pre-shame; only surface mismatch on submit attempt) | `disabled` | enabled |
| **typing-match** | `lower(input) === lower(user.email)` | No alert | **enabled** (`bg-destructive`) | enabled |
| **submitting** | submit clicked, request in flight | No alert | `disabled` + label `Deleting‚Ä¶` | `disabled` |
| **success** | 204 returned | Dialog closes synchronously; `authStore.logout()` + `navigate('/login')`; no in-dialog success state needed | n/a | n/a |
| **error: 400 mismatch** | server-side rejection (defence-in-depth ‚Äî client-side gate should prevent this) | `<Alert variant="destructive">Confirmation email does not match.</Alert>` | re-enabled if input still matches; otherwise stays disabled | enabled |
| **error: 429** | RateLimitError | `<Alert variant="destructive">Too many requests. Try again in {N}s.</Alert>` | `disabled` for the countdown duration (UI-09) | enabled |
| **error: other** | ApiClientError or generic | `<Alert variant="destructive">Could not delete account. Try again.</Alert>` | re-enabled (after `submitting ‚Üí false`) if input still matches | enabled |

Disabled‚Üíenabled transition is **purely client-side** based on `confirmEmail.trim().toLowerCase() === user.email.toLowerCase()` ‚Äî no debounce, no async validation. Enables on every keystroke once match holds.

### `LogoutAllDialog` ‚Äî single-confirm

| State | Trigger | Visual | Confirm button |
|-------|---------|--------|-----------------|
| **idle** | Dialog opened | No alert | enabled |
| **submitting** | Confirm clicked | No alert | `disabled` + label `Signing out‚Ä¶` |
| **success** | 204 returned | Dialog closes; logout + redirect | n/a |
| **error: 429** | RateLimitError | `<Alert variant="destructive">Rate limited. Try again in {N}s.</Alert>` | `disabled` for countdown |
| **error: other** | ApiClientError or generic | `<Alert variant="destructive">Could not sign out. Try again.</Alert>` | re-enabled |

### `UpgradeInterestDialog` ‚Äî three-state form

| State | Trigger | Visual | Submit button |
|-------|---------|--------|----------------|
| **idle** | Dialog opened | textarea visible, no alert | enabled (textarea is optional ‚Äî empty submission allowed) |
| **submitting** | Submit clicked | textarea `disabled` | `disabled` + label `Sending‚Ä¶` |
| **success** | 501 returned (swallowed as success) OR 200 | textarea + submit replaced by success Alert + copy + auto-close 2s timer | n/a |
| **error: 429** | RateLimitError | `<Alert variant="destructive">Too many requests. Try again in {N}s.</Alert>` above textarea | `disabled` for countdown |
| **error: other** | ApiClientError (excluding 501 which is success here) | `<Alert variant="destructive">Could not send. Try again.</Alert>` above textarea | re-enabled |

### `AccountPage` hydration state machine

| State | Trigger | Render |
|-------|---------|--------|
| **loading** | mount: `useEffect` triggers `fetchAccountSummary()` and component has `summary === null && error === null` | Three skeleton cards (Profile / Plan / Danger Zone) ‚Äî each is a `<Card class="p-6">` containing 1 placeholder heading line + 2 muted lines via `<div class="h-4 w-32 rounded bg-muted">` style; render time should be sub-200ms typically; **no spinner, no "Loading‚Ä¶" text** |
| **error** | hydration request failed (any non-401 error; 401 redirects via apiClient) | Single `<Card class="p-6">` showing `<Alert variant="destructive"><AlertDescription>Could not load account. {retry button}</AlertDescription></Alert>` + `<Button variant="outline" size="sm" onClick={refresh}>Try again</Button>` |
| **empty** | n/a ‚Äî every authenticated user has a profile; this state cannot occur | (no UI defined; defensive code paths fall through to error state) |
| **ready** | summary fetched | three cards rendered as per Layout section |

`apiClient.get('/api/account/me')` in `authStore.refresh()` runs on app boot with `suppress401Redirect: true` (probe-style ‚Äî CONTEXT.md ¬ß83); a 401 there silently leaves `user === null` and `RequireAuth` redirects to `/login?next=` as today (no double-redirect race).

`AccountPage` itself reads `useAuthStore(s => s.user)`; if `user === null` after RequireAuth gate (impossible by contract) it renders the error card. Otherwise it issues a **second** `fetchAccountSummary()` call to get fresh `trial_started_at` + `token_version` for that page only ‚Äî keeps store hydration cheap and page detail loud.

## Accessibility Contract

- All three dialogs inherit Radix `Dialog` semantics (focus trap, Escape closes, restored focus on close, `role="dialog"` + `aria-labelledby` + `aria-describedby` automatic via `DialogTitle` + `DialogDescription`).
- Email input in `DeleteAccountDialog` declares `autoComplete="off"` to suppress browser autofill on a destructive form (XSS / shoulder-surf hardening).
- Destructive buttons declare `aria-label` only if icon-only; with text label as specified, no aria-label needed.
- Plan-tier badge has `role="status"` is **not** required ‚Äî it is informational text inside a heading-labeled card; sufficient context exists.
- Color contrast: shadcn neutral default already exceeds WCAG AA at all variant pairings in the contract. Destructive `oklch(0.577 0.245 27.325)` on white meets AA for `text-sm`/`text-lg` (OKLCH lightness 0.577 + chroma 0.245 ‚Üí ~4.6:1 on white per shadcn audit ‚Äî sufficient for non-large text).
- Keyboard: Tab order in `DeleteAccountDialog` = email input ‚Üí Cancel ‚Üí Delete account; in `LogoutAllDialog` = Cancel ‚Üí Sign out everywhere; in `UpgradeInterestDialog` = textarea ‚Üí Cancel ‚Üí Send. shadcn `DialogFooter` reverses on `< sm` (`flex-col-reverse`) which keeps destructive button at bottom on mobile (lower thumb risk) ‚Äî accept this default.
